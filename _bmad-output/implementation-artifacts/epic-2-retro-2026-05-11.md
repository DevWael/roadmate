## Successes

### 1. TripEndEvent Decoupled Architecture

Story 2.3 refactored TripDetector from directly saving completed trips to emitting `TripEndEvent` via SharedFlow. This decoupled the state machine from persistence and paid dividends in Stories 2.4 (crash recovery with INTERRUPTED status) and 2.7 (gap handling with different finalization paths) — each just needed a different event type, no architectural change.

**Evidence:** TripDetector, TripRecorder, and CrashRecoveryManager all operate on event streams with no direct coupling between state machine and persistence.

### 2. Zero Production Incidents Despite High Complexity

Trip detection state machine, GPS gap handling, crash recovery journal, power loss protection, Room DB migration — all without a single production incident. The review process caught every critical issue before merge.

**Evidence:** 76 review findings, 61 patched immediately, 0 incidents reported.

### 3. Room DB Migration Pattern Established

Story 2.7's v1→v2 migration adding `isGapBoundary` to TripPoint established the migration pattern that all future schema changes will follow. Migration is tested and documented.

**Evidence:** `RoadMateDatabase` version incremented, migration class with test coverage.

### 4. Dual Constructor Pattern for Testable Coroutines

Story 2.1 established the pattern of `@Inject` constructor creating default scope with a primary constructor accepting injectable scope. This became the standard for GpsTracker, TripDetector, TripRecorder, CrashRecoveryJournal — every coroutine-based component.

**Evidence:** Every core component uses this pattern. Test code injects `TestScope` via primary constructor.

### 5. Haversine Distance with Accuracy Filtering

The distance calculation pipeline correctly excludes low-accuracy points (>50m) from distance calculation while still persisting them for route completeness. This prevents garage drift from inflating trip distance.

**Evidence:** HaversineCalculator + TripRecorder accuracy filtering tested across Stories 2.3 and 2.7.

### 6. Defensive Code Review Maintained

80% immediate fix rate across 76 findings — up from 76% in Epic 1. Review caught race conditions, state cleanup bugs, boundary errors, and platform-specific issues across every story.

**Evidence:** Stories 2.4 and 2.6 had 13 and 12 findings respectively — all critical issues patched.

## Challenges

### 1. State Cleanup Completeness (5/7 Stories)

`resetState()`, `gracefulShutdown()`, `resetGapState()` repeatedly missed cleaning up mutable fields — `gapTimeoutJob`, `firstHighSpeedTimestamp`, `isGapMode`, `postGapRecovery`, timer jobs. This appeared in Stories 2.1, 2.2, 2.3, 2.4, 2.7. Each time we said "we'll be more careful" and review found more missed fields.

**Root cause:** No structured approach to state cleanup. Developers write cleanup for the "happy path" state and miss edge-case accumulated state.

**Impact:** Orphaned coroutines, stale state after transitions, potential resource leaks.

### 2. Coroutine Test Infrastructure Gaps

Stories 2.1, 2.2, 2.3, and 2.7 all fought with coroutine testing: `UncompletedCoroutinesError`, `TurbineTimeoutCancellationException`, scheduler isolation causing test hangs. The `advanceTimeBy` infrastructure for testing timer-based code (10s flush, 30s journal, 5min gap timeout) was identified as missing in every affected story but never built.

**Root cause:** No upfront investment in test coroutine infrastructure. Each story worked around it instead of solving it once.

**Impact:** Timer-based ACs verified architecturally but not by actual tests. Confidence gap on flush intervals, gap timeouts, journal timing.

## Review Findings Synthesis

### Pattern: State Cleanup Completeness (5/7 stories)

`resetState()` and cleanup methods miss mutable fields. Appeared in Stories 2.1, 2.2, 2.3, 2.4, 2.7. Pattern: developer writes cleanup for the primary state, forgets edge-case accumulated fields (gap timeout jobs, timestamps, flags).

**Recommendation:** Adopt a "cleanup checklist" pattern — every class with mutable state must declare all mutable fields in a single block and `resetState()` must reset every one.

### Pattern: Race Conditions in Shared State (4/7 stories)

Async operations racing with state reads: trip creation vs `endTrip` (2.2), `ensureTripLoaded` vs first flush (2.3), `GapRecoveredEvent` vs DrivingState flow (2.7), journal write/read timing (2.4).

**Recommendation:** Use `Mutex` or `StateFlow` as single-writer pattern for shared mutable state. Document which thread owns which state.

### Pattern: Off-by-One / Boundary Errors (3/7 stories)

Accuracy threshold `>` vs `>=` (2.7), float vs double precision in thresholds (2.2), speed boundary values at exact thresholds.

**Recommendation:** Boundary values must have explicit tests at the threshold value itself (not just above/below).

### Pattern: Time Source Mixing (3/7 stories)

`System.currentTimeMillis()` used alongside injectable `Clock` in Stories 2.2, 2.4, 2.5. Partially fixed but persists in pre-existing code (VehicleRepository.addToOdometer).

**Recommendation:** Ban `System.currentTimeMillis()` in production code. Use injectable `Clock` everywhere.
